## 1. 问题陈述

公司目前由一名财务人员使用 Word 和 Excel 手工维护学员信息、缴费记录、退款记录及相关凭证。

当前数据量较小，主要风险不是查询速度或多人协作，而是缴费、退款金额被手工录错。系统首版的目标是主动阻止金额错误进入正式账务，并确保错误发生后可以追溯、作废和更正。

## 2. 目标用户与首版范围

### 目标用户

- 一名公司内部财务人员
- 暂不建设复杂角色权限和多人审批流程

### 首版包含

- 学员档案管理
- 培训期数文本字段
- 缴费记录管理
- 退款记录管理
- 缴费和退款多附件上传、预览、下载
- 金额二次确认
- 正式记录作废与重新录入
- 操作记录

### 首版不包含

- 开班签名
- 独立培训单据模块
- Excel 导出
- 多人复核审批
- 批量导入历史 Word、Excel 数据

## 3. 已确认前提

1. 首版服务于一名财务人员，不建设复杂权限和多人审批。
2. 核心目标是减少缴费、退款金额录错，而不是提升批量处理效率。
3. 缴费、退款采用“草稿 -> 金额二次确认 -> 正式入账”流程。
4. 正式入账后的记录不可直接修改或删除。
5. 正式记录发现错误时，只能填写原因后作废，再重新录入正确记录。
6. 作废记录永久保留，不能从操作界面物理删除。
7. 首版包含学员档案、缴费、退款、附件和操作记录。

## 4. 推荐方案

采用不可变账务流水方案。

缴费和退款在正式入账前允许编辑草稿。正式入账后，金额、日期、支付方式、关联缴费、备注和附件均被冻结。错误记录通过“作废并重新录入”更正，系统保留原记录、作废原因、作废时间和新记录之间的关系。

这比普通增删改查多一步确认，但能直接解决当前最严重的金额录错问题，并提供清晰的事后追溯能力。

## 5. 核心业务流程

### 5.1 新增学员

1. 财务填写姓名、手机号、身份证号、地址、培训期数。
2. 系统校验必填字段和格式。
3. 系统校验身份证号唯一；手机号允许多个学员共用。
4. 财务确认后保存学员档案。

### 5.2 缴费入账

1. 财务在学员详情页新增缴费草稿。
2. 填写收款金额、返点金额、转培训成本金额、缴费日期、支付方式、备注，并上传零个或多个附件；实际收款金额由系统自动计算。
3. 系统保存草稿，草稿允许继续编辑或删除。
4. 财务点击“确认入账”。
5. 系统展示独立确认页面，突出显示：
   - 学员姓名和手机号
   - 收款金额
   - 返点金额
   - 转培训成本金额
   - 实际收款金额，由系统自动计算
   - 缴费日期
   - 支付方式
   - 附件数量
6. 确认页展示草稿原金额供核对；财务仍需对照凭证再次输入收款金额、返点金额和转培训成本金额。
7. 系统重新计算实际收款金额，仅在三个输入金额分别与草稿完全一致时允许正式入账。
8. 正式入账后记录和附件被冻结。

### 5.3 退款入账

1. 财务在学员详情页新增退款草稿。
2. 必须关联某笔已正式入账且未作废的缴费。
3. 填写退款金额、退款日期、退款原因，并上传零个或多个附件。
4. 系统保存草稿。
5. 财务点击“确认入账”，系统显示独立确认页面。
6. 财务必须再次输入退款金额。
7. 系统校验金额一致后正式入账。
8. 正式入账后记录和附件被冻结。

### 5.4 作废与重新录入

1. 财务在正式记录上点击“作废”。
2. 系统要求填写作废原因，并再次确认。
3. 系统将记录标记为已作废，不删除原记录和附件。
4. 已作废记录不计入有效缴费、退款合计。
5. 系统提供“基于此记录重新录入”操作，复制非金额字段到新草稿。
6. 新草稿必须重新经过金额二次确认才能入账。

## 6. 功能设计

### 6.1 学员列表

- 展示姓名、手机号、身份证号脱敏值、培训期数、有效实际收款合计、有效退款合计。
- 支持按姓名、手机号、培训期数查询。
- 培训期数支持精确和模糊搜索。
- 支持分页和基础排序。
- 点击学员进入详情页。

### 6.2 学员详情

页面顶部展示学员基本信息和有效收支汇总。

详情页包含以下标签：

- 基本信息
- 缴费记录
- 退款记录
- 操作记录

缴费、退款列表必须明确展示记录状态：草稿、已入账、已作废。

### 6.3 学员删除

首版不允许物理删除已有财务记录的学员。

- 无任何缴费或退款记录的学员可以删除。
- 存在草稿或正式财务记录的学员只能停用。
- 停用学员保留全部历史记录，不能再新增缴费或退款。

### 6.4 附件

- 缴费和退款均支持一次上传多个文件。
- 支持常见图片和 PDF 在线预览；其他文件提供下载。
- 附件显示原始文件名、大小、上传时间。
- 草稿附件可以新增或删除。
- 正式入账后附件不可新增、替换或删除。
- 作废记录的附件继续保留。
- 单个文件大小、单次上传数量和允许类型需配置。

### 6.5 操作记录

操作记录至少保存：

- 操作类型
- 操作对象和记录编号
- 操作前状态
- 操作后状态
- 操作时间
- 操作人
- 作废原因
- 原记录与重新录入记录的关联关系

## 7. 业务规则

### 金额规则

- 收款金额和实际收款金额必须大于 0。
- 返点金额和转培训成本金额必须大于或等于 0。
- 金额最多保留两位小数。
- 前端和后端必须使用十进制金额处理，不使用浮点数。
- 实际收款金额固定等于“收款金额 - 返点金额 - 转培训成本金额”，由后端自动计算，不能手动修改。
- 确认缴费入账时必须对照凭证再次输入收款金额、返点金额和转培训成本金额，不能从原输入框自动复制。
- 后端必须再次校验三个输入金额分别与草稿完全一致。

### 退款规则

- 退款必须关联某笔已正式入账且未作废的缴费。
- 一笔缴费允许分多次退款，累计有效退款不得超过该笔缴费的实际收款金额。
- 存在有效退款的缴费不可作废，必须先作废关联退款。

### 状态规则

| 状态 | 可编辑 | 可删除 | 可作废 | 计入汇总 |
|---|---:|---:|---:|---:|
| 草稿 | 是 | 是 | 否 | 否 |
| 已入账 | 否 | 否 | 是 | 是 |
| 已作废 | 否 | 否 | 否 | 否 |

## 8. 数据结构建议

- 姓名
- 手机号
- 身份证号，唯一
- 地址
- 培训期数
- 状态：正常、停用
- 创建时间、更新时间

### 缴费

- 缴费编号
- 学员编号
- 收款金额
- 返点金额
- 转培训成本金额
- 实际收款金额
- 缴费日期
- 支付方式
- 备注
- 状态
- 作废原因、作废时间
- 来源记录编号
- 创建时间、入账时间

### 退款

- 退款编号
- 学员编号
- 关联缴费编号，必填
- 金额
- 退款日期
- 退款原因
- 状态

- 来源记录编号

- 操作时间

## 9. 安全与数据保护

- 身份证号在列表默认脱敏，只在必要页面查看完整值。
- 附件下载和预览必须经过登录校验，不能使用永久公开链接。
- 数据库和附件需要每日自动备份。
- 备份必须定期验证可以恢复。
- 所有正式入账、作废和停用操作必须写入操作记录。
- 系统需设置会话超时，避免财务电脑离开后被他人操作。

## 10. 备选方案

### 方案 A：轻量表单系统

普通增删改查，提交前弹窗确认金额。

- 工作量：S
- 风险：中
- 优点：上线最快，操作步骤少。
- 缺点：正式记录可被修改，无法可靠追溯历史。

### 方案 B：不可变账务流水，已选择

草稿经过金额二次确认后入账，正式记录只能作废重录。

- 工作量：M
- 风险：低
- 优点：直接防止金额误录，历史记录完整。
- 缺点：更正错误需要额外步骤。

### 方案 C：双录入核对

同一金额录入两次，系统比对一致后入账。

- 工作量：M
- 风险：低
- 优点：防错能力强。
- 缺点：单人操作负担较高，长期可能被规避。

## 11. 验收标准

### 核心验收

- 财务无法在任一缴费输入金额二次输入不一致时正式入账。
- 财务无法在退款金额二次输入不一致时正式入账。
- 正式缴费和退款记录无法直接修改或删除。
- 作废记录仍可查看全部字段、附件和作废原因。
- 作废记录不计入有效收支合计。
- 重新录入的记录能够追溯到原作废记录。
- 退款关联缴费时，系统阻止累计退款超过已确认的退款基准金额。
- 学员详情页可完整查看该学员的缴费、退款、附件和操作记录。

### 文件验收

- 一次可上传多个附件。
- 图片和 PDF 可预览。
- 正式入账后附件不能被修改或删除。
- 删除草稿时，其附件一并删除。

### 数据保护验收

- 身份证号在学员列表中脱敏。
- 备份任务可执行，并完成一次恢复演练。
- 正式入账、作废、停用操作均可在操作记录中查到。

## 12. 上线策略

### 第一阶段：试运行

- 仅录入新发生的学员、缴费和退款。
- 原 Word、Excel 文件继续作为历史档案，不立即批量迁移。
- 财务连续使用一周，记录错误提示、操作阻塞和遗漏字段。

### 第二阶段：正式使用

- 修复试运行中发现的问题。
- 明确系统作为新业务记录的唯一入口。
- Word、Excel 转为只读历史档案。

### 后续版本候选

- 开班签名
- 独立培训单据
- Excel 导出
- 历史数据导入
- 多人复核
- 财务报表

## 13. 开放问题

这些问题不阻塞首版设计，但需在工程计划阶段确认：

1. 支付方式的固定选项有哪些？
2. 附件允许的文件类型、单文件大小和单次上传数量上限是多少？
3. 系统部署在公司内网还是公网？
4. 操作人是否只需要一个财务账号，还是还需要管理员账号？

## 14. 成功标准

上线后首月：

- 因键入错误导致的正式入账金额错误为 0；实际收款金额全部由系统按固定公式计算。
- 所有正式记录的修改均通过作废重录完成。
- 任一学员的缴费、退款和凭证可在 1 分钟内完整找到。
- 财务能够独立完成新增学员、缴费入账、退款入账和作废重录。

## 15. 下一步任务

让财务人员使用纸面或简单原型，完整演练一次以下流程，并记录卡点：

“新增学员 -> 新增缴费草稿 -> 金额二次确认 -> 正式入账 -> 发现错误 -> 作废 -> 重新录入正确金额”。

这次演练的结果用于确认字段、按钮文案和作废流程，再进入工程设计。

## 16. 本次讨论中确认的判断

- “数据量比较小”意味着首版不需要为批量处理或复杂报表优化。
- “由一个财务人员手动处理”意味着多人审批不是首版重点。
- “缴费或退款金额录错”被明确为后果最严重的问题。
- “财务自行二次确认”确定了单人二次确认流程。
- 已确认正式入账后禁止直接修改和删除。
