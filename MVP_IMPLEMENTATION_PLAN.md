## 1. 执行摘要

MVP 服务公司内部财务人员，将学员、缴费、退款和凭证从 Word/Excel 迁移到可追溯系统。第一目标不是报表，而是减少金额录错，并保证正式记录无法静默修改。

核心方案是不可变账务流水：

- 缴费、退款先保存草稿，再对照凭证进行独立确认。
- 缴费实际收款金额由系统计算：`收款金额 - 返点金额 - 转培训成本金额`。
- 退款必须关联有效缴费，累计退款不得超过该缴费的实际收款金额。
- 存在有效退款的缴费不可作废，必须先作废关联退款。
- 正式记录不可修改或删除，只能填写原因后作废并重新录入。
- 操作日志仅追加，记录所有关键状态变化。

## 2. 已确认业务前提

| # | 前提 | 结论 |
|---|---|---|
| P1 | 实际收款金额计算 | 系统自动计算，不允许手动修改 |
| P2 | 退款上限 | 使用关联缴费的实际收款金额 |
| P3 | 退款关联 | MVP 中退款必须关联一笔有效缴费；允许分多次退款 |
| P4 | 缴费作废 | 存在有效退款时禁止作废缴费 |
| P5 | 防错方式 | 正式入账确认页必须展示凭证和学员身份，财务对照凭证重新输入金额 |
| P6 | 手机号和身份证号 | 手机号允许重复；身份证号标准化后唯一 |
| P7 | 正式记录 | 不可修改、不可删除，只能作废重录 |

## 3. CEO / 产品审查

### 3.1 正确的问题

当前最值得解决的是“金额录错后难发现、难追溯”，不是复杂报表或多人协作。不可变流水与凭证对照确认直接覆盖该风险。

二次输入只防止键入不一致，不能证明凭证本身被正确理解。因此 MVP 的确认页必须让财务同时看到凭证、学员身份、业务公式和待确认金额。

### 3.2 CEO 双重审查共识

| 维度 | 独立审查 | Codex 审查 | 共识 |
|---|---|---|---|

| 金额规则 | 原文不可实施 | 原文不可实施 | 用户已确认固定公式 |
| 退款规则 | 原文不可实施 | 原文不可实施 | 用户已确认强关联与上限 |
| 作废规则 | 关联边界缺失 | 关联边界缺失 | 用户已确认先处理退款 |
| 历史数据 | 双重事实来源风险 | 双重事实来源风险 | MVP 明确切换日和口径提示 |
| 日终对账 | 建议加入 MVP | 建议加入 MVP | 延后为首个增强项 |

### 3.5 MVP 不包含

- 开班签名与独立培训单据
- Excel 导出、完整财务报表
- 历史 Word/Excel 批量迁移
- 日终资金对账与账期关闭
- 多人审批、组织级权限体系
- OCR 自动识别金额
- 非图片/PDF格式的在线预览

### 3.6 上线口径

- 设定明确的系统切换日期。
- 系统中的合计均标记“仅包含系统切换日后的记录”。
- 旧 Word/Excel 保持只读，作为历史资料来源。
- MVP 上线后新业务只允许进入系统。

## 4. 设计审查

### 4.1 信息架构

```text
登录
  └─ 首页工作队列
      ├─ 待确认缴费
      ├─ 待确认退款
      ├─ 上传失败记录
      ├─ 最近作废记录
      └─ 学员列表
      ├─ 新增学员
      └─ 学员详情
          ├─ 基本信息
          ├─ 缴费记录
          │   ├─ 新增/编辑草稿
          │   ├─ 凭证对照确认
          │   ├─ 查看正式记录
          │   └─ 作废/基于原记录重录
          ├─ 退款记录
          │   ├─ 选择关联缴费
          │   ├─ 新增/编辑草稿
          │   ├─ 凭证对照确认
          │   └─ 作废/基于原记录重录
          └─ 操作记录
```

### 4.2 页面与关键交互

| 页面 | 第一视觉重点 | 关键操作 | 危险操作保护 |
|---|---|---|---|
| 学员列表 | 搜索与学员身份 | 新增、查看、停用 | 不提供物理删除；停用需原因 |
| 学员详情 | 身份信息与切换日后收支汇总 | 新增缴费、退款 | 停用前二次确认 |
| 缴费草稿 | 三个输入金额与自动计算结果 | 保存、上传凭证、进入确认 | 实际收款只读 |
| 缴费确认 | 凭证、身份、草稿金额、空白金额输入框 | 正式入账 | 展示草稿金额供核对；任一输入不一致则阻止 |
| 退款草稿 | 关联缴费、剩余可退金额 | 保存、上传凭证 | 超额退款实时阻止 |
| 作废对话框 | 影响与原因 | 确认作废 | 必填原因；存在有效退款时阻止 |

### 4.3 交互状态覆盖

所有列表、表单和附件区域必须定义：

- 加载中：固定布局骨架，不允许重复提交。
- 空状态：说明当前无记录，并提供明确的下一步操作。
- 校验错误：字段旁显示问题、原因和修复方法。
- 上传中：显示每个文件进度；部分失败允许单独重试。
- 保存成功：显示记录状态与下一步操作。
- 并发冲突：提示记录已变化，刷新后重新确认。
- 会话过期：保留未提交表单内容，登录后恢复草稿。
- 权限不足：不显示危险操作入口，服务端仍强制校验。
- 入账成功：跳转只读正式记录页，显示记录编号、时间和状态；重复提交返回“该草稿已入账”。
- 未保存离开：阻止离开并提示；正式退出时清除浏览器临时输入。
- 作废预览：展示记录编号、金额、汇总影响、关联记录、附件保留和不可撤销说明。

页面级状态矩阵必须分别覆盖：首页工作队列、学员列表、学员详情、草稿表单、确认页和附件区。每个页面明确 `loading / empty / error / success / partial / conflict` 状态及允许操作。

### 4.4 可访问性与响应式

- 首版以桌面端为主，最低支持 1280px；手机端允许只读查看，不承诺完整录入体验。
- 所有操作可使用键盘完成，焦点顺序与视觉顺序一致。
- 状态不能只依赖颜色，必须同时有文本标签。
- 危险按钮使用明确动词，如“作废缴费”，不使用含糊的“确认”。
- 金额使用等宽数字和千分位展示，输入框中不使用千分位。
- 满足 WCAG 2.1 AA；错误摘要可跳转到字段；对话框焦点锁定并在关闭后恢复。
- 验收覆盖 `1280x720`、`1366x768`、`1920x1080` 和浏览器 200% 缩放。
- 窄于桌面录入断点时隐藏写操作并说明原因，金额不得截断。

### 4.5 独立确认页面规范

- 桌面端采用“左侧凭证预览，右侧确认表单”布局。
- 展示学员身份、培训期数、业务公式和草稿原金额，金额确认输入框保持空白。
- 财务面对凭证填写三个空白金额字段，系统计算实际收款后与草稿比较。
- 不一致时仅指出不匹配字段，不显示原值；用户只能返回草稿修改。
- 退款确认同时展示关联缴费编号、日期、实际收款、已退款、剩余可退和入账后余额。
- 正式入账按钮位于页面末尾，提交期间禁用。

## 5. 工程方案

### 5.1 推荐技术架构

为快速上线并减少运维面，采用单体 Web 应用：

- 后端：FastAPI
- 数据库：PostgreSQL
- ORM 与迁移：SQLAlchemy 2 + Alembic
- 服务端模板：Jinja2
- 轻交互：HTMX + 少量原生 JavaScript
- 样式：项目内 CSS，不引入大型前端构建链
- 文件存储：S3 兼容对象存储；本地开发使用 MinIO
- 身份认证：服务端 Session Cookie
- 测试：pytest + Playwright
- 部署：Docker Compose，反向代理负责 TLS

选择原因：项目是低并发内部工具，服务端渲染单体能以最小复杂度交付完整、可测试的业务流程。账务正确性集中在同一个事务边界内。

### 5.3 数据模型

#### users

- `id`, `username`, `password_hash`, `role`, `is_active`
- 角色仅包含 `finance` 与 `admin`

#### students

- `id`, `name`, `phone`, `id_number_ciphertext`, `id_number_hash`
- `address`, `training_period_text`, `status`
- `created_at`, `updated_at`, `version`
- `id_number_hash` 唯一；手机号不唯一

#### payments

- `id`, `student_id`, `receipt_amount`, `rebate_amount`, `transfer_cost_amount`
- `actual_received_amount`，由后端计算并持久化
- `student_name_snapshot`, `student_id_masked_snapshot`, `training_period_snapshot`
- `payment_date`, `payment_method`, `note`, `status`
- `source_payment_id`, `void_reason`, `voided_at`
- `confirmed_at`, `created_by`, `confirmed_by`, `version`

#### refunds

- `id`, `payment_id`
- `amount`, `refund_date`, `reason`, `status`
- `source_refund_id`, `void_reason`, `voided_at`
- `confirmed_at`, `created_by`, `confirmed_by`, `version`

#### attachments

- `id`, `storage_key`, `storage_version`, `status`
- `original_name`, `content_type`, `size_bytes`, `sha256`
- `created_by`, `created_at`

通过 `payment_attachments` 和 `refund_attachments` 关联表建立真实外键，不使用多态 `owner_type + owner_id`。

#### audit_events

- `id`, `actor_id`, `action`, `entity_type`, `entity_id`
- `before_snapshot`, `after_snapshot`, `reason`, `created_at`
- 只允许插入，不提供更新和删除接口
- 快照使用字段白名单；身份证仅记录脱敏值或“已变更”标记

#### idempotency_keys

- `actor_id`, `operation`, `key`, `request_hash`
- `result_entity_type`, `result_entity_id`, `response_status`, `expires_at`
- `(actor_id, operation, key)` 唯一；同一键对应不同请求内容时返回冲突

### 5.4 数据库约束与状态机

- 金额统一使用人民币 `NUMERIC(14,2)`，不接受三位小数。
- 收款、返点、转培训成本均大于或等于 0，实际收款必须大于 0。
- 数据库 `CHECK` 强制实际收款等于固定公式。
- 退款金额必须大于 0；退款通过 `payment_id` 推导学员。
- 正式记录保存入账时的学员姓名、脱敏身份证号和培训期数快照。
- 重录链禁止自引用、环路和一个原记录分叉到多个正式替代记录。

```text
DRAFT --confirm--> POSTED --void--> VOIDED
  |
  +--abandon--> ABANDONED

Attachment:
UPLOADING --> READY --> FROZEN
    |           |
    +--> FAILED +--> DELETE_PENDING --> DELETED
```

每条状态转换必须定义允许角色、前置条件、锁、可修改字段、审计事件、幂等响应和失败后的状态。

### 5.5 事务、锁与不可变规则

- 正式入账、附件元数据冻结、审计日志写入必须处于同一数据库事务。
- 对象存储不参与数据库事务：上传先写临时不可变对象并校验哈希，再将数据库附件状态标记为 `READY`。
- 入账只允许关联全部为 `READY` 的附件；事务内将附件元数据标记为 `FROZEN`。
- 后台清理任务处理孤儿对象和 `DELETE_PENDING`，定期一致性扫描比较对象版本与哈希。
- 确认入账接口必须重新读取草稿并检查版本号，避免旧页面覆盖新修改。
- 服务端重新计算实际收款金额，不信任客户端传入值。
- 退款确认、退款作废、缴费作废均先 `SELECT payment FOR UPDATE`，再在锁内重新计算有效退款和校验状态。
- 事务锁顺序固定为“缴费 -> 退款 -> 附件”；定义锁等待超时和可重试错误。
- 正式记录更新接口仅允许状态从 `POSTED` 转为 `VOIDED`。
- 数据库约束与服务层规则同时存在，不能只依赖前端。
- 确认入账接口使用幂等键；网络超时或重复提交不能产生第二条正式记录。

## 6. 错误与救援登记

| 场景 | 用户看到什么 | 系统行为 | 恢复路径 |
|---|---|---|---|
| 四类金额不匹配 | 明确标出不匹配字段 | 阻止入账，不改变草稿 | 对照凭证重新输入 |
| 实际收款小于等于 0 | 显示公式与结果错误 | 阻止保存/入账 | 修正三个输入金额 |
| 超额退款 | 显示剩余可退金额 | 阻止确认 | 降低退款或选择正确缴费 |
| 有退款的缴费被作废 | 显示有效退款列表 | 阻止作废 | 先作废关联退款 |
| 附件上传部分失败 | 每个文件单独显示失败原因 | 已成功文件保留 | 单独重试失败文件 |
| 重复身份证号 | 指向已有学员 | 阻止新增 | 查看已有学员或修正号码 |
| 并发修改草稿 | 提示草稿已变化 | 阻止旧版本确认 | 刷新并重新确认 |
| 数据库不可用 | 显示操作未保存 | 整个事务回滚 | 服务恢复后重试 |
| 对象存储不可用 | 显示附件不可用 | 不允许带缺失凭证入账 | 存储恢复后重试 |
| 会话过期 | 提示重新登录 | 不执行提交 | 登录后恢复草稿 |

## 7. 失败模式登记

| 风险 | 严重性 | 预防 | 验证 |
|---|---:|---|---|
| 金额第一次即抄错 | 高 | 确认页同时展示凭证与身份 | 财务演练 + E2E |
| 实际收款公式被绕过 | 高 | 后端计算 + DB 检查 | 单元与集成测试 |
| 并发退款导致超额 | 高 | 事务锁 + 重新计算 | 并发集成测试 |
| 正式记录被修改 | 关键 | 无更新接口 + DB 触发/权限限制 | 集成测试 |
| 作废后汇总仍计入 | 高 | 查询统一过滤有效状态 | 集成测试 |
| 附件与记录不一致 | 高 | 上传状态与确认前校验 | 集成与 E2E |
| 审计日志缺失 | 高 | 与状态变化同事务写入 | 集成测试 |
| 敏感信息泄露 | 关键 | 加密、脱敏、授权下载 | 安全测试 |
| 备份无法恢复 | 关键 | 定期恢复演练 | 上线门禁 |

## 8. 性能、运维与安全

### 性能

- 数据量小，不引入缓存。
- 学员查询对姓名、手机号、培训期数建立索引。
- 缴费、退款按 `student_id + status + date` 建复合索引。
- 列表查询使用数据库聚合，避免逐学员查询收支合计。
- 附件使用对象存储直传或流式上传，不将完整文件读入内存。

### 安全

- 密码使用 Argon2id 哈希。
- Session Cookie 设置 `HttpOnly`、`Secure`、`SameSite=Lax`。
- 所有表单启用 CSRF 防护。
- 身份证号加密存储，并用不可逆哈希执行唯一性校验。
- 身份证唯一校验使用服务端密钥参与的 HMAC；加密密钥纳入备份、轮换和恢复演练。
- 附件验证扩展名、MIME、大小和文件签名；下载使用短时授权。
- 服务端对每个写操作执行角色校验。
- 审计日志和备份禁止财务角色删除。
- 登录限流；登录成功后轮换 Session；备份加密；附件默认下载并限制内联执行。

### 运维

- 健康检查覆盖应用、数据库和对象存储。
- 结构化日志包含请求 ID、用户 ID、实体 ID和错误码，不记录身份证明文。
- 每日备份 PostgreSQL 与对象存储清单。
- 备份包含 PostgreSQL、附件对象本体、加密密钥和部署配置。
- 目标：RPO 不超过 24 小时，RTO 不超过 4 小时；保留 30 天并保存异地副本。
- 上线前完成一次从备份恢复到空环境的演练，并校验附件哈希、账务汇总和审计链。
- 数据库迁移前自动备份；迁移采用向后兼容变更，并提供失败回滚手册。

## 10. 分阶段开发任务

### 阶段 0：业务规则样例与项目基线

#### T0.1 收集并固化金额样例

工作：

- 由财务提供至少 5 个脱敏真实案例，覆盖普通缴费、返点、转培训成本、分次退款和作废重录。
- 将案例写成业务规则测试夹具。

验收标准：

- 每个案例都包含三个输入金额、系统计算实际收款、退款上限和预期状态变化。
- 财务负责人书面确认所有案例结果正确。
- 自动化测试可读取这些夹具。

#### T0.2 建立项目骨架与本地环境

工作：

- 初始化 Git、FastAPI、PostgreSQL、MinIO、pytest、Playwright 和 Docker Compose。
- 固定依赖版本并提交锁文件。
- 建立命令契约表中的所有阶段 0 命令。

验收标准：

- 新环境执行 `make doctor && make bootstrap && make dev` 后在 10 分钟内打开登录页。
- `make bootstrap` 可重复执行，不重复创建账号或破坏数据。
- 分层测试命令在干净环境通过，失败时输出产物路径。
- `.env.example` 不包含真实密钥。
- 开发、测试、生产配置差异有明确矩阵；生产不能使用开发默认密钥或账号。

#### T0.3 实现数据库约束、状态机和并发原型

验收标准：

- 数据库约束拒绝非法金额、公式不一致、非法状态转换和正式记录业务字段修改。
- 两个并发退款确认不能超额；退款确认与缴费作废并发时只有合法操作成功。
- 幂等键重复请求返回原结果；同一键不同请求体返回冲突。
- 这些测试在业务页面开发前全部通过。

#### T0.4 实现附件生命周期原型

验收标准：

- 附件完整经历 `UPLOADING / READY / FROZEN / DELETE_PENDING / FAILED`。
- 数据库提交失败和对象存储失败均不会产生不可追溯的不一致状态。
- 一致性扫描可识别孤儿对象、缺失对象和哈希不一致。

### 阶段 1：身份认证、学员与审计基线

#### T1.1 实现登录与角色

验收标准：

- 财务和管理员账号可以登录、退出。
- 未登录用户无法访问任何业务页面和附件。
- 财务无法执行管理员限定操作。
- 会话过期后写操作不执行，未提交草稿可恢复。

#### T1.2 实现学员档案

验收标准：

- 可新增、编辑、搜索和查看学员。
- 手机号允许重复。
- 标准化身份证号重复时阻止新增，并定位已有学员。
- 列表身份证号脱敏。
- MVP 不提供任何学员物理删除入口；错误学员也只能填写原因后停用。
- 停用学员不可新增缴费或退款，不可确认现有草稿；历史记录仍可查看。

#### T1.3 实现只追加审计日志

验收标准：

- 学员新增、编辑、停用均写入审计日志。
- 日志包含操作人、时间、实体、前后快照。
- 财务账号无法修改或删除日志。

### 阶段 2：缴费草稿、附件与正式入账

#### T2.1 实现缴费金额模型和草稿

验收标准：

- 财务可录入收款、返点、转培训成本。
- 实际收款由后端按公式计算并只读展示。
- 实际收款小于等于 0 时无法保存或入账。
- 所有金额使用十进制两位小数。

#### T2.2 实现多附件上传与授权访问

验收标准：

- 草稿可上传、下载、预览多个图片/PDF附件。
- 每个附件独立显示上传结果，失败可单独重试。
- 未授权用户无法访问附件。
- 删除草稿时删除其附件；正式记录附件不可删除。

#### T2.3 实现凭证对照确认与正式入账

验收标准：

- 确认页同时显示凭证、姓名、脱敏身份证号、手机号、培训期数、公式和草稿原金额，金额确认输入框保持空白。
- 财务必须重新输入三个原始金额，系统重新计算实际收款。
- 任一金额不匹配、附件不可用或草稿版本变化时阻止入账。
- 成功入账后记录和附件冻结，并写入审计日志。
- 重复提交、网络超时重试不能产生重复正式记录。

#### T2.4 实现缴费作废与重录

验收标准：

- 作废必须填写原因并再次确认。
- 作废记录仍可查看且不计入汇总。
- 可基于作废记录创建新草稿，新记录可追溯原记录。
- 存在有效退款时缴费作废被阻止。
- 作废前展示对汇总、关联记录和附件的影响，以及不可撤销说明。

### 阶段 3：退款闭环

#### T3.1 实现退款草稿和缴费关联

验收标准：

- 退款必须选择有效缴费。
- 页面显示实际收款、已退金额和剩余可退金额。
- 可对同一缴费创建多笔退款草稿。
- 停用学员不能新增退款。

#### T3.2 实现退款确认与并发保护

验收标准：

- 财务对照退款凭证重新输入金额。
- 超过剩余可退金额时无法入账。
- 两个并发退款确认不能导致累计退款超额。
- 正式退款不可修改或删除，并写入审计日志。
- 确认页展示关联缴费编号、日期、实际收款、已退款、剩余可退和退款后余额。

#### T3.3 实现退款作废与缴费作废联动

验收标准：

- 退款作废后恢复缴费可退余额。
- 作废退款不计入退款合计。
- 所有关联、作废和重录链条可追溯。
- 最后一笔有效退款作废后，原缴费才允许作废。

### 阶段 4：查询、状态覆盖与安全加固

#### T4.0 实现首页工作队列

验收标准：

- 首页展示待确认缴费、待确认退款、上传失败记录和最近作废记录。
- 每项显示学员、记录类型、创建时间和停留时长，并可直接进入处理页面。
- 队列为空、部分加载失败和全部加载失败时都有明确状态。

#### T4.1 完成学员列表与详情汇总

验收标准：

- 支持姓名、手机号、培训期数搜索及分页排序。
- 详情页显示切换日后的有效实际收款、退款与净额。
- 草稿、已入账和已作废状态清晰区分。
- 汇总不包含草稿和作废记录。

#### T4.2 完成所有交互状态和错误恢复

验收标准：

- 加载、空、错误、成功、部分上传失败、会话过期和并发冲突均有可操作界面。
- 写操作按钮提交期间不可重复点击。
- 所有服务端错误使用稳定错误码，并给出修复提示。
- 页面级状态矩阵覆盖全部核心页面。
- 未保存离开、表单恢复和并发冲突不会静默丢失用户输入。

#### T4.3 完成安全与隐私检查

验收标准：

- 通过认证、授权、CSRF、附件访问、输入校验和敏感日志检查。
- 身份证号数据库明文不可见。
- 财务账号无法修改正式记录、审计日志或备份。
- 依赖漏洞扫描无高危未处理项。

### 阶段 5：上线、试运行与恢复演练

#### T5.1 建立部署与备份

验收标准：

- 可从版本化镜像部署到目标环境。
- TLS、会话安全配置和健康检查生效。
- PostgreSQL 与附件每日自动备份。
- 从备份恢复到空环境并验证关键记录与附件可访问。
- 部署 Runbook 明确构建、备份、迁移、启动、健康检查、冒烟测试和回滚步骤。
- 镜像固定 digest；迁移失败或冒烟测试失败时可按 Runbook 回滚。
- `make backup/restore/verify-restore` 完成一次空环境演练并记录 RPO/RTO。

#### T5.2 财务端到端验收

验收标准：

- 财务独立完成：新增学员、缴费入账、退款入账、作废退款、作废缴费、重录。
- 使用真实脱敏案例时，系统计算和退款余额全部正确。
- 任一学员的新系统记录和凭证可在 1 分钟内找到。
- 财务确认确认页信息足以发现录入错误。

#### T5.3 一周试运行与切换

验收标准：

- 明确系统切换日期并在界面显示汇总口径。
- 一周内所有新业务只录入系统。
- 阻塞问题全部修复；非阻塞问题进入 `TODOS.md`。
- 完成正式上线批准记录。

## 15. 跨阶段主题

1. **可信度来自外部证据，而非重复输入。** CEO 和设计审查均要求确认页突出凭证与学员身份。
2. **账务规则必须由后端强制。** 产品和工程审查均要求固定公式、强关联退款和不可变正式记录。
